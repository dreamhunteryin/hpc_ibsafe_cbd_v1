### R1 — No root new files
Do not create new files in repository root. Use `src/` modules.

### R2 — Output locations
Generated outputs must go to `data/exports/` or `output/` (never root).

### R3 — No new .md unless requested
Do not create additional markdown files unless the user explicitly requests.

### R4 — No duplicate implementations / single source of truth
Avoid multiple implementations for the same functionality. Extend existing code.

### R5 — No duplicate filenames
Do not create `*_v2.py`, `*_new.py`, `enhanced_*` or similar “versioned” duplicates. Extend/modify existing files.

### R6 — Avoid copy-paste
Extract shared logic into reusable functions/modules instead of duplicating code.

### R7 — Configurable values
Avoid hard-coded values that should be configurable. Use config/environment variables.

### R8 — Tooling discipline
Do not use shell `find/grep/cat/head/tail/ls`. Use the platform tools: Read / LS / Grep / Glob.

### R9 — Git checkpointing
After each meaningful milestone, run the standard save routine: **add → commit → push** to `main`.

### R10 — Long-running / background work
Use Task Agents only for I/O-heavy, long-running, or background operations that may exceed ~120 seconds.
If you estimate the task execution will exceed ~120 seconds, use agent for background excutive

If the user explicitly requests background execution

## 5) PROJECT STRUCTURE (REFERENCE)

> Goal: One repo supports two pipelines:
> (A) Credit Card Statement Analyzer (existing)
> (B) Automatic Journal Search (PubMed → Abstract dump → zh-TW formatted summaries → exports)

### 5.1 Top-level directories (no new root files; see R1)

src/
  ccsa/                       # existing credit-card analyzer domain
  pubmed/                     # PubMed E-utilities client + query builder
  journal_search/             # orchestration: month/journal runs, splitting, QC hooks
  llm/                        # OpenAI API wrapper, prompt templates, batching, caching
  parsing/                    # MEDLINE/XML parsing -> normalized records
  exports/                    # docx/markdown/html exporters (docx primary)
  ui/                         # optional (Streamlit/React hooks) - keep thin
  utils/                      # logging, retry, rate-limit, io, typing

data/
  inputs/                     # optional manual inputs (e.g., pmid list, debug samples)
  cache/
    pubmed/                   # raw efetch responses (xml/medline), keyed by hash
    llm/                      # response cache (jsonl), keyed by hash+model+prompt_ver
  exports/                    # user-facing outputs (docx, jsonl, csv)

output/
  runs/                       # run artifacts: logs, qc reports, intermediate snapshots
  temp/                       # ephemeral; safe to purge

tests/
  unit/
  integration/

scripts/
  run_journal_month.py        # CLI entry (thin wrapper; calls src/journal_search)
  run_ccsa.py                 # existing entry (if present)

### 5.2 Single-source-of-truth data model

Normalized article record (internal):
- pmid (str), doi (str|None)
- journal (str), year (int), dp (date|None), epdat (date|None), edat (date|None)
- title (str), abstract (str|None)
- authors (list[str]), affiliations (list[str]|None)
- links: pubmed_url, doi_url
- provenance: query_hash, fetch_timestamp, source="pubmed"

All downstream steps (QC, summarise, export) consume this normalized record only.

## 6) CORE WORKFLOWS (HIGH-LEVEL)

1) Query
- In: journal + month + anchor + topic
- Out: pubmed query + query_hash

2) Search
- In: query
- Out: pmid set (split if needed)

3) Fetch
- In: pmids
- Out: raw fetch cache + normalized records.jsonl

4) Screen (optional)
- In: records.jsonl
- Out: kept.jsonl + excluded.jsonl

5) Summarize
- In: kept.jsonl
- Out: summaries.jsonl (zh-TW, non-verbatim)

6) Export
- In: kept.jsonl + summaries.jsonl
- Out: docx + kept_list.csv
